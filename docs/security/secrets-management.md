# Secrets Management — SAKN

**Date**: 2026-05-20
**Scope**: All application secrets — generation, storage, validation, and rotation procedures.

## 1. Overview

| Secret | Purpose | Generated By | Enforced At |
|--------|---------|-------------|-------------|
| `SECRET_KEY` | HMAC key for email hashing in security event logs | Operator (CSPRNG) | `config.py` validates length in production |
| `POSTGRES_PASSWORD` | PostgreSQL authentication for the `sakn` user | Operator (CSPRNG) | `docker-compose.yml` fail-fast (`:?`) |
| `REDIS_PASSWORD` | Redis authentication via `--requirepass` | Operator (CSPRNG) | `docker-compose.yml` fail-fast (`:?`) |
| `SMTP_PASSWORD` | SMTP relay authentication for email delivery | Provider-supplied | None (SMTP is optional) |

**General principles**:
- All secrets are generated via CSPRNG (`secrets.token_urlsafe(32)` — 256 bits of entropy).
- Secrets are never hardcoded in source code.
- Secrets are stored in `.env` files, which are listed in `.gitignore`.
- Production Docker Compose uses `${VAR:?error message}` fail-fast expansion to
  refuse startup when a required secret is missing or empty.
- The `.env.example` file at the repo root provides a template with placeholder
  values and generation instructions.

## 2. SECRET_KEY

### Purpose
- HMAC-SHA256 key for session token hashing (`security/tokens.py:hash_token()`),
  preventing token forgery if both DB and Redis are compromised (ADR-007, fixes
  audit finding M-3).
- HMAC-SHA256 key for hashing email addresses in `SecurityEventLog` entries
  (prevents plaintext email storage in logs).
- Code references: `src/backend/app/security/tokens.py:hash_token()`,
  `src/backend/app/services/auth_service.py:_hash_email_for_log()`.

### Generation
```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```
Produces a 43-character base64url string with 256 bits of entropy.

### Validation
`src/backend/app/config.py` applies the following checks:
- In **production** (`ENVIRONMENT=production`): rejects the default value
  `change-me-in-production-use-at-least-32-bytes-base64` and any key shorter
  than 32 characters.
- In **development**: accepts shorter keys (convenience for local dev).

### Storage
- `.env` file at the repo root (loaded by Pydantic `BaseSettings`).
- Docker Compose `environment:` block passes it to the backend container.
- Must **never** be committed to version control.

### Rotation Procedure

1. **Generate** a new key:
   ```bash
   python -c "import secrets; print(secrets.token_urlsafe(32))"
   ```
2. **Update** `SECRET_KEY` in the `.env` file.
3. **Restart** the backend:
   ```bash
   docker compose restart backend
   ```
4. **Invalidate all sessions** (old CSRF tokens were generated under the old
   key context): follow the procedure in
   [Incident Response — Invalidate All Sessions](./incident-response.md#41-invalidate-all-sessions).
5. **Note**: email hashes in existing `SecurityEventLog` rows are permanently
   bound to the old key. Queries that hash an email with the new key will not
   match old log entries. Accept this as archival data; old logs expire after
   the configured retention period (default 90 days).

## 3. POSTGRES_PASSWORD

### Purpose
Authentication for the `sakn` PostgreSQL user (superuser within the `sakn`
database).

### Generation
```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

### Validation
`docker-compose.yml` line 38:
```yaml
POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?POSTGRES_PASSWORD is required}
```
Docker Compose refuses to start if the variable is unset or empty.

### URL Encoding
`src/backend/app/config.py` assembles the `DATABASE_URL` from individual
`POSTGRES_*` components. The password is URL-encoded via
`urllib.parse.quote()` to handle special characters (`@`, `:`, `/`, `%`, etc.):
```
postgresql+asyncpg://sakn:<url-encoded-password>@postgres:5432/sakn
```

### Storage
- `.env` file (root). Never committed.
- Passed to both `postgres` and `backend` containers via Docker Compose `environment:`.

### Rotation Procedure

1. **Generate** a new password:
   ```bash
   python -c "import secrets; print(secrets.token_urlsafe(32))"
   ```
2. **Update** `POSTGRES_PASSWORD` in `.env`.
3. **Change** the password in PostgreSQL:
   ```bash
   docker compose exec postgres psql -U sakn -c "ALTER USER sakn WITH PASSWORD 'new_password';"
   ```
4. **Restart** the backend:
   ```bash
   docker compose restart backend
   ```
5. **Verify** the backend connects successfully:
   ```bash
   curl -s http://localhost:8000/health
   # Expected: {"status":"ok"}
   ```
6. **Securely delete** the old password from any backups or password managers.

## 4. REDIS_PASSWORD

### Purpose
Redis authentication enforced via the `--requirepass` server option.

### Generation
```bash
python -c "import secrets; print(secrets.token_urlsafe(32))"
```

### Validation
`docker-compose.yml` lines 43 and 95:
```yaml
command: redis-server --appendonly yes --requirepass ${REDIS_PASSWORD:?REDIS_PASSWORD is required}
```
Docker Compose refuses to start if the variable is unset or empty.

The `REDIS_URL` includes the password in the connection string:
```
redis://:${REDIS_PASSWORD}@redis:6379/0
```
The colon before the password is intentional per RFC 3986 (empty username,
password present).

### Storage
- `.env` file (root). Never committed.
- Passed to the `redis` container via `command:` and to the `backend` container
  via `environment:`.

### Rotation Procedure

1. **Generate** a new password.
2. **Update** `REDIS_PASSWORD` in `.env`.
3. **Update** Redis at runtime (no-downtime option):
   ```bash
   docker compose exec redis redis-cli -a "$OLD_REDIS_PASSWORD" CONFIG SET requirepass "$NEW_REDIS_PASSWORD"
   ```
   Alternatively, restart Redis (brief downtime):
   ```bash
   docker compose restart redis
   ```
4. **Restart** the backend:
   ```bash
   docker compose restart backend
   ```
5. **Verify**:
   ```bash
   curl -s http://localhost:8000/health
   ```

## 5. SMTP_PASSWORD

### Purpose
Authentication for the SMTP relay used to send email verification and password
reset messages.

### Generation
Provided by the SMTP service operator (e.g., SendGrid, Mailgun, AWS SES,
self-hosted Postfix). Not generated by SAKN.

### Validation
No fail-fast validation. SMTP is **optional** — when `SMTP_HOST` is empty or
unset, the application operates without email delivery.

### Storage
- `.env` file. Never committed.
- Related variables: `SMTP_HOST`, `SMTP_PORT`, `SMTP_USERNAME`, `SMTP_PASSWORD`,
  `SMTP_FROM`.

### Rotation Procedure
1. Follow your SMTP provider's password/key rotation process.
2. Update `SMTP_PASSWORD` in `.env`.
3. Restart the backend:
   ```bash
   docker compose restart backend
   ```

## 6. The `.env.example` Pattern

The repo root contains `.env.example` — a template with all required
environment variables, placeholder values (`CHANGEME`), and inline
documentation. Operators should:

1. Copy the template:
   ```bash
   cp .env.example .env
   ```
2. Replace placeholder values with CSPRNG-generated secrets.
3. Verify `.env` is listed in `.gitignore` (it is — confirmed in the repo root
   `.gitignore`).

This pattern directly addresses audit findings:
- **C-4**: Default PostgreSQL password — prevented by fail-fast in production compose.
- **H-7**: Default SECRET_KEY in dev — prevented by fail-fast validation in production config.

The `.env.example` file itself **is** committed to version control (it contains
no secrets, only documentation and placeholders).
