# Audit de sécurité — SAKN

- **Date** : 2026-05-18
- **Auditeur** : revue statique automatisée (Claude Opus 4.7)
- **Périmètre** : code applicatif (backend FastAPI, frontend React/Vite), infrastructure (Docker Compose, Caddy, Nginx), configuration et documentation de sécurité.
- **Hors périmètre** : test de pénétration dynamique, fuzzing, audit RGPD processus, scan SCA des dépendances, exécution effective des images Docker.

---

## Contexte

SAKN (Swiss Army Knife for Network Engineers) expose des outils réseau (ping / traceroute / DNS lookup / SSL viewer) via une interface web. L'application sert à la fois des visiteurs anonymes (avec rate-limit IP) et des utilisateurs authentifiés (rôles `visitor`, `authenticated`, `administrator`).

Le présent rapport décrit les faiblesses, vulnérabilités et écarts aux bonnes pratiques identifiés à la date de l'audit. Chaque finding dispose d'un identifiant stable (`C-x`, `H-x`, `M-x`, `L-x`) utilisé comme ancre et référencé dans les issues GitHub correspondantes.

---

## Résumé exécutif

**Posture globale** : la base est de bonne qualité (Argon2id, CSRF double-submit, énumération-safe, ORM, segmentation Docker, CSP/HSTS partiels) mais **plusieurs défauts bloquants** rendent la base actuelle inadaptée à une mise en production en l'état :

- Suppression de compte côté frontend **totalement fictive** (le backend fonctionne, mais le client ne l'appelle pas).
- Caddy n'écoute **qu'en HTTP** (`:80`) — pas de TLS configuré.
- Crash garanti dans le job `cleanup_unverified_accounts` (`NameError: datetime`).
- Mot de passe Postgres par défaut faible (`sakn`) et Redis sans authentification.
- Dossier `docs/security/` (avant ce rapport) vide alors que `CLAUDE.md` l'impose.

## Findings

### Critiques

#### C-1 — Suppression de compte fictive côté frontend
- **Fichier** : `src/frontend/src/pages/account/AccountDeletePage.tsx:17-25`
- **Description** : `handleDelete()` exécute `await new Promise(r => setTimeout(r, 500))` puis `logout()` + `navigate("/login")`. Aucun appel API. Le backend implémente pourtant correctement `DELETE /account` (`src/backend/app/api/v1/endpoints/account.py:81-128`) avec vérification du mot de passe, anonymisation des logs et suppression des sessions.
- **Impact** : conformité RGPD (droit à l'effacement) compromise. Les utilisateurs croient leur compte supprimé alors qu'il subsiste en base.
- **Recommandation** : remplacer la simulation par `api("/account", { method: "DELETE", body: { password } })`, gérer l'erreur retournée et ne déclencher `logout()` qu'après succès.

#### C-2 — Reverse proxy Caddy en HTTP seul, pas de TLS
- **Fichier** : `config/Caddyfile:1`
- **Description** : la directive est `:80 { ... }` — Caddy n'écoute que sur le port 80. Aucun bloc `https://` ni gestion ACME. Les headers HSTS/CSP sont émis sur HTTP, donc ignorés par les navigateurs, et tout le trafic transite en clair.
- **Impact** : interception MITM, vol de cookies session/CSRF, downgrade trivial. Les flags `Secure` et `__Host-` sur les cookies sont sans effet faute de TLS.
- **Recommandation** : remplacer par un site bloc nommé (`example.com { ... }`) géré par Caddy via ACME, ou `https://...`. Ajouter un bloc `http://...` qui redirige `301` vers HTTPS. Ajouter `preload` à HSTS une fois la chaîne validée.

#### C-3 — `NameError` garanti dans le job de nettoyage des comptes non vérifiés
- **Fichier** : `src/backend/app/main.py:228-249`
- **Description** : `cleanup_unverified_accounts` importe localement `from datetime import timedelta, timezone` (ligne 232) puis utilise `datetime.now(timezone.utc)` (ligne 236). `datetime` n'est **jamais** importé, ni au module ni dans la fonction. Le job apscheduler lèvera `NameError` à chaque exécution (cron quotidien 03:30).
- **Impact** : aucun nettoyage des comptes non vérifiés. Accumulation indéfinie en base, contournement potentiel des contraintes de vérification email.
- **Recommandation** : `from datetime import datetime, timedelta, timezone` à la ligne 232 et ajouter une couverture de test unitaire pour ce scheduler.

#### C-4 — Mot de passe PostgreSQL par défaut faible
- **Fichier** : `docker-compose.yml:35,73`
- **Description** : `POSTGRES_PASSWORD=${POSTGRES_PASSWORD:-sakn}` — si l'opérateur oublie la variable d'environnement, Postgres démarre avec un mot de passe trivial connu publiquement.
- **Impact** : si Postgres devient accessible (mauvaise config réseau, exposition de port, sortie d'isolation Docker), accès complet aux données utilisateurs.
- **Recommandation** : retirer la valeur par défaut, ajouter un fail-fast au `startup.sh` si `POSTGRES_PASSWORD` non défini, documenter dans `.env.example`.

### Élevés

#### H-1 — Redis sans authentification
- **Fichier** : `docker-compose.yml:85-97`, `docker-compose.dev.yml:32-44`
- **Description** : `redis-server --appendonly yes` ; aucun `--requirepass`. Redis stocke les sessions et les compteurs de rate-limit. Bien que sur réseau Docker `sakn-internal`, c'est une dépendance forte à l'isolation réseau.
- **Impact** : un side-channel (autre conteneur compromis, escape, mauvaise config) donne accès à toutes les sessions actives et permet leur vol.
- **Recommandation** : `--requirepass ${REDIS_PASSWORD}`, propager `REDIS_URL=redis://:password@redis:6379/0`. Activer `protected-mode` et envisager TLS Redis si trafic cross-host.

#### H-2 — WebSocket : permission contournable sur erreur DB (fail-open)
- **Fichier** : `src/backend/app/api/v1/endpoints/tools.py:155-211`
- **Description** : tout le bloc de vérification (tool activé + permission rôle) est encapsulé dans un `try / except Exception: pass` (lignes 210-211). Toute erreur (DB indisponible, query qui throw, etc.) saute la vérification et tombe directement dans l'acceptation du WebSocket et l'exécution de l'outil. Le commentaire `# Allow execution if DB check fails` confirme l'intention — c'est un fail-open.
- **Impact** : un attaquant peut potentiellement déclencher un dysfonctionnement DB (saturation pool, contention) pour bypasser la matrice RBAC.
- **Recommandation** : `await websocket.close(code=4503)` sur exception ; logger l'événement. Le fail-closed est la bonne politique pour un check d'autorisation.

#### H-3 — WebSocket sans validation CSRF ni Origin
- **Fichier** : `src/backend/app/api/v1/endpoints/tools.py:141-229`
- **Description** : le WebSocket lit le cookie de session via parsing manuel mais n'effectue aucune vérification CSRF ni de l'header `Origin`. Le middleware CSRF (`BaseHTTPMiddleware`) ne s'applique pas aux WebSockets.
- **Impact** : Cross-Site WebSocket Hijacking. Un site malveillant peut ouvrir un WS authentifié avec les cookies de la victime et exécuter des outils en son nom.
- **Recommandation** : valider `websocket.headers["origin"]` contre une allowlist (`settings.CORS_ORIGINS`) avant `accept()`. Optionnellement, exiger un token CSRF en sous-protocole ou query param.

#### H-4 — Énumération d'emails via les logs de sécurité
- **Fichier** : `src/backend/app/services/auth_service.py:104,199,308`
- **Description** : la stratégie « enumeration-safe » côté client (ADR-002) est solide, mais côté serveur les événements `registration_duplicate`, `login_failed_no_user` et `password_reset_request_no_user` stockent l'email en clair dans `details` de `SecurityEventLog`.
- **Impact** : énumération de comptes massive en cas de fuite de logs ; stockage de PII dans les logs (RGPD).
- **Recommandation** : hasher l'email avec un secret applicatif avant de logger (`hmac_sha256(SECRET_KEY, email)`), ou ne pas inclure l'email du tout pour ces événements précis.

#### H-5 — `ilike` sans échappement des wildcards SQL
- **Fichier** : `src/backend/app/api/v1/endpoints/admin_users.py` (recherche email administrateur)
- **Description** : `User.email.ilike(f"%{search}%")` ne neutralise pas `%` et `_` dans l'input. Pas une SQLi (SQLAlchemy paramètre la valeur), mais permet l'usage de wildcards en abus.
- **Impact** : usage abusif de l'endpoint admin pour énumération via wildcards.
- **Recommandation** : échapper `\\`, `%`, `_` puis utiliser `ilike(..., escape="\\")`.

#### H-6 — Frontend Nginx servi sans aucun header de sécurité
- **Fichier** : `src/frontend/nginx.conf:1-23`
- **Description** : aucun `add_header` pour CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy. Caddy en frontal pallie en production, mais si Nginx est exposé directement (dev, debug, mauvaise topologie), aucune protection.
- **Impact** : clickjacking, MIME sniffing, dépendance forte au reverse proxy.
- **Recommandation** : dupliquer les headers du Caddyfile dans `nginx.conf` (au minimum X-Content-Type-Options, X-Frame-Options, Referrer-Policy).

#### H-7 — Default `SECRET_KEY` connu dans `docker-compose.dev.yml`
- **Fichier** : `docker-compose.dev.yml:8`
- **Description** : `SECRET_KEY=${SECRET_KEY:-dev-secret-key-not-for-production-use-only}`. Si ce fichier est déployé tel quel (par accident), la clé qui signe les CSRF est publique.
- **Impact** : forge de cookies/CSRF, contournement potentiel de protections.
- **Recommandation** : retirer la valeur par défaut (fail-fast). En dev local, exiger un `.env` non versionné.

#### H-8 — Fallback DB silencieux vers SQLite en production
- **Fichier** : `src/backend/app/main.py:42-70`
- **Description** : si Postgres est injoignable, l'app bascule silencieusement sur `sqlite+aiosqlite:///./sakn.db` au sein du conteneur. En production : données écrites localement, perdues au redémarrage, incohérence entre instances, absence d'alerting.
- **Impact** : perte d'intégrité des données, incidents non détectés.
- **Recommandation** : conditionner le fallback à `ENVIRONMENT == "development"` ; sinon lever et faire crasher le pod (fail-fast).

#### H-9 — `/tools/{tool_name}/dns-servers` ne respecte pas le flag `enabled`
- **Fichier** : `src/backend/app/api/v1/endpoints/tools.py:89-117`
- **Description** : endpoint public qui renvoie les serveurs DNS pour n'importe quel `tool_name`, qu'il soit activé ou non, sans vérifier la permission du rôle.
- **Impact** : information disclosure sur la configuration d'outils désactivés.
- **Recommandation** : filtrer par `ToolModule.enabled == True` ET vérifier la permission `RoleToolPermission`.

### Moyens

#### M-1 — Headers de sécurité backend limités à `/api/*`
- **Fichier** : `src/backend/app/middleware/security_headers.py:14-24`
- **Description** : le middleware n'applique CSP/X-Frame-Options/HSTS qu'à `/api/`. `/health` et tout autre chemin n'ont aucun header.
- **Recommandation** : appliquer à toutes les routes, gérer les exceptions via une liste.

#### M-2 — CSP avec `style-src 'unsafe-inline'`
- **Fichier** : `config/Caddyfile:3`, `src/backend/app/middleware/security_headers.py:16`
- **Recommandation** : utiliser des nonces ou hashes pour les styles inline générés par React/Vite, ou auditer le bundle pour supprimer le besoin.

#### M-3 — Hash de token de session en SHA-256 simple (sans pepper)
- **Fichier** : `src/backend/app/security/tokens.py:10-12`
- **Description** : tokens générés via CSPRNG 256 bits, donc le risque brute-force est théorique. Mais l'absence de pepper applicatif rend l'inversion plus rapide en cas de fuite simultanée DB+Redis.
- **Recommandation** : `hmac.new(SECRET_KEY.encode(), token.encode(), "sha256")`. Migration progressive via double-écriture.

#### M-4 — `/tools/available-for/{role}` accepte une chaîne arbitraire
- **Fichier** : `src/backend/app/api/v1/endpoints/tools.py:29-51`
- **Description** : pas de validation d'enum. Permet de cartographier les rôles en DB.
- **Recommandation** : valider `role` contre `Literal["visitor","authenticated","administrator"]`.

#### M-5 — `/health` divulgue l'état infrastructure sans authentification
- **Fichier** : `src/backend/app/main.py:321-345`
- **Description** : renvoie `{database, redis}` avec `ok/unavailable`. Donne à un attaquant la cartographie de l'infra et la temporalité des incidents.
- **Recommandation** : un `/health` minimal (200 OK simple) pour LB externe, un `/health/full` interne (bind interne ou auth header).

#### M-6 — Parsing manuel de cookie WS non robuste
- **Fichier** : `src/backend/app/api/v1/endpoints/tools.py:128-138`
- **Description** : `cookies.split(";")` + `startswith` est fragile face à des valeurs encodées, des espaces, des cookies doublons.
- **Recommandation** : utiliser `http.cookies.SimpleCookie` pour le parsing.

#### M-7 — `docs/security/` vide alors que `CLAUDE.md` l'impose
- **Fichier** : `docs/security/` (corrigé partiellement par le présent rapport)
- **Description** : pas de threat model, pas de procédure de rotation des secrets, pas d'IR.
- **Recommandation** : créer `threat-model.md`, `secrets-management.md`, `incident-response.md`.

#### M-8 — Verrou anti-brute-force basé uniquement sur l'utilisateur
- **Fichier** : `src/backend/app/services/auth_service.py:26-32,212-227`
- **Description** : les paliers (5/10/15/20) s'incrémentent sur `user.failed_login_attempts`. Un attaquant peut DoS un compte précis, ou faire du credential stuffing sans déclencher de protection IP.
- **Recommandation** : combiner avec un rate-limit IP strict (déjà partiellement présent), ajouter un compteur par IP au-delà duquel un captcha est requis.

#### M-9 — Tests E2E `.mjs` à la racine du frontend (hors framework)
- **Fichier** : `src/frontend/test-*.mjs` (14 fichiers à la racine)
- **Description** : scripts Playwright ad-hoc à la racine, hors du dossier `e2e/` et hors de la config `playwright.config.ts`.
- **Recommandation** : migrer dans `e2e/` ou supprimer ; ajouter `*.mjs` à la racine au `.gitignore` du frontend si scripts jetables.

#### L-1 — Message d'erreur incorrect pour mot de passe trop long
- **Fichier** : `src/backend/app/security/password.py:37`
- Renvoie `"errors.password_too_short"` pour un mot de passe `len > 128`. Bug fonctionnel, faible impact sécurité.

#### L-2 — Chemin `setcap` traceroute à vérifier
- **Fichier** : `src/backend/Dockerfile:20`
- `setcap cap_net_raw+ep /usr/bin/traceroute.db` — le suffixe `.db` est inhabituel ; à valider dans l'image construite (peut être le binaire Debian legacy ou un chemin invalide).

#### L-3 — Vite dev proxy en HTTP
- **Fichier** : `src/frontend/vite.config.ts`
- Acceptable en local, à ne pas reproduire en staging.

#### L-4 — Cookie `lang=` sans flag Secure
- **Fichier** : `src/frontend/src/i18n/i18n.ts:45`
- Pas sensible mais incohérent avec la politique cookies du reste de l'app.

#### L-5 — Headers manquants : Permissions-Policy, COOP, COEP
- Aucune émission. Recommandé pour limiter les capacités du navigateur et l'isolation cross-origin.

#### L-6 — Playwright `trace: "on-first-retry"` par défaut
- **Fichier** : `src/frontend/playwright.config.ts`
- Peut capturer des données sensibles dans les traces de tests en échec. Conditionner à `process.env.CI`.

#### L-7 — Aucun scan SCA en CI
- Pas de `pip-audit`, `npm audit` ni `osv-scanner` configurés. Recommandé pour détecter les CVE des dépendances.

## Points forts confirmés

À conserver / valoriser :

- ✅ **Argon2id** pour les mots de passe avec paramètres OWASP-compatibles.
- ✅ **Énumération-safe** sur registration / login / password reset (ADR-002 implémenté).
- ✅ Brute-force lockout multi-paliers (5/10/15/20).
- ✅ CSRF double-submit (cookie + header `X-CSRF-Token`), validation explicite dans chaque endpoint d'état.
- ✅ Sessions HttpOnly + SameSite=Lax + `__Host-` quand HTTPS.
- ✅ FastAPI **docs/redoc/openapi désactivés** en production.
- ✅ Cap Docker `NET_RAW`/`NET_ADMIN` correctement scopé au seul backend, container en `USER sakn` (non-root).
- ✅ Réseau Docker `sakn-internal` marqué `internal: true` (pas de sortie internet).
- ✅ SQLAlchemy ORM partout — aucune raw SQL.
- ✅ Audit logs (`AuditLog`, `SecurityEventLog`, `ToolExecutionLog`) modélisés.
- ✅ `.env` et `sakn.db` correctement listés dans `.gitignore`.
- ✅ `secrets.token_urlsafe(32)` pour génération de tokens, `secrets.compare_digest` pour comparaisons constantes.

## Limitations de l'audit

- **Audit statique uniquement** : pas de test de pénétration, pas de fuzzing, pas de DAST.
- **Pas d'exécution de l'image Docker** : `L-2` non confirmé à l'exécution.
- **Dépendances non scannées** : pas de comparaison avec une base CVE.
- **Pas d'analyse de flux RGPD** : traitement code, pas processus métier.
- **Pas d'analyse SIEM** : qualité de la détection (alerting, corrélation) non évaluée.
- **Tests applicatifs non exécutés** : la suite `src/backend/tests/` couvre-t-elle les chemins d'authentification ? À vérifier.

## Verdict

**Accept with reservations** — la base est de bonne qualité, mais les findings critiques (C-1 à C-4) et plusieurs findings élevés (H-1 à H-3, H-7, H-8) doivent être corrigés **avant tout déploiement public**.
